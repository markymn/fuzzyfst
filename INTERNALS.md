# FuzzyFST Internals

Implementation notes on codegen, pruning, and performance decisions.

## Myers' bit-parallel step(): sbb/adc branchless codegen

The `step()` function updates the Levenshtein distance after consuming one FST character. The critical dist update code:

```cpp
uint32_t dist = state.dist;
uint64_t last_bit = 1ULL << (m - 1);
if (Ph & last_bit) dist += 1;
if (Mh & last_bit) dist -= 1;
```

GCC 13.2 with `-O3 -march=native` compiles this to:

```asm
cmpq    $1, %rsi         ; CF = (Ph & last_bit == 0)
sbbl    $-1, %r10d       ; dist += CF ? 0 : 1  (i.e., dist++ when Ph & last_bit)
cmpq    $1, %r8          ; CF = (Mh & last_bit == 0)
adcl    $-1, %r10d       ; dist += CF ? -1 : 0  (i.e., dist-- when Mh & last_bit)
```

This is **fully branchless** and better than the `cmov` we initially targeted:

- **sbb/adc**: single-cycle, no data dependency between the two updates. The CPU's carry flag from `cmp` feeds directly into `sbb`/`adc`. Total: 4 instructions, 2 cycles (the two pairs can execute in parallel).
- **cmov**: would require a test + cmov per branch (4 instructions) but with a data dependency on the FLAGS register that prevents parallel execution of the two pairs.

The rest of `step()` (bitvector operations Xv, Xh, Ph, Mh, new_Pv, new_Mv) compiles to straight-line ALU ops with no branches.

## can_match(): full DP column reconstruction is a correctness requirement

The `can_match()` function determines whether any extension of the current target prefix can yield a match within `max_dist`. It reconstructs the minimum value across the entire implicit DP column D[0..m] from the Pv/Mv bitvectors:

```cpp
bool LevenshteinNFA::can_match(const LevenshteinState& state) const {
    if (state.dist <= max_dist) return true;  // Fast path: bottom cell already within budget.

    uint32_t val = state.dist;
    for (uint32_t j = query_len; j > 0; --j) {
        uint32_t bit = j - 1;
        if (pv & (1ULL << bit)) {
            if (--val <= max_dist) return true;  // Early exit.
        } else if (mv & (1ULL << bit)) {
            val++;
        }
    }
    return false;
}
```

### Bottom-cell-only checking is incorrect

Our original plan described full-column reconstruction as an optional phase-6 optimization. Experimental measurement proved this wrong: **checking only the bottom cell D[m] is a correctness bug, not a performance tradeoff.**

A naive `can_match` that only checks `state.dist <= max_dist` (the bottom cell D[m]) silently drops valid results by over-pruning. The mechanism:

1. Myers' bitvectors encode the full DP column D[0..m] implicitly. D[m] is the edit distance between the complete query and the current target prefix. But interior cells D[j] for j < m represent the edit distance between query[0..j-1] and the target prefix.

2. When D[m] > max_dist but some D[j] <= max_dist (j < m), the target prefix is within distance of a *prefix* of the query. Future target characters can still match the remaining query suffix query[j..m-1], potentially bringing D[m] within budget for a longer target word.

3. Bottom-cell-only checking prunes this state because D[m] > max_dist, discarding all words reachable through this FST subtree. Those words include valid matches that would have been found if traversal continued.

**Example:** For query "algoritm" (m=8) with max_dist=1, after consuming target prefix "al", the DP column might be D[0]=2, D[1]=1, D[2]=0, D[3]=1, ..., D[8]=6. D[8]=6 > 1, so bottom-cell-only prunes. But D[2]=0, meaning "al" exactly matches "al" (query[0..1]). If the FST contains "algorithm", the remaining target characters "gorithm" would match query[2..7]="goritm" within distance 1. Bottom-cell-only misses this.

### Measured impact

We ran 100 representative queries (misspellings generated by deleting the middle character of dictionary words) against the full 370,105-word FST, comparing full-column reconstruction against bottom-cell-only:

| Pruning strategy | d=1 visited | d=1 results | d=1 miss rate | d=2 visited | d=2 results | d=2 miss rate |
|-----------------|-------------|-------------|---------------|-------------|-------------|---------------|
| Full column (correct) | 215,737 | 499 | — | 1,829,090 | 8,007 | — |
| Bottom cell only (incorrect) | 4,489 | 157 | **69%** | 48,971 | 3,910 | **51%** |

Bottom-cell-only misses **69% of valid d=1 results** and **51% of valid d=2 results**. The 48x reduction in visited nodes at d=1 is not efficiency — it is the search failing to explore paths that contain valid matches. This was confirmed by cross-referencing against a brute-force oracle over the full dictionary.

Full-column reconstruction is not optional. It is the standard correct pruning condition for Levenshtein NFA intersection with an FST.

### Assembly: can_match loop

The loop compiles to `bt` (bit test) + conditional branch with early exit. Unlike `step()`, the loop inherently requires branching (early exit on first cell <= max_dist). The fast-path check `state.dist <= max_dist` avoids entering the loop entirely for ~50-70% of calls (any state where the bottom cell is already within budget).

## Distance-2 pruning behavior

See README.md for the full analysis. In summary: at d=2, 0% of depth-1 transitions are pruned because any single non-matching character produces a column with min value = 1 <= 2. This is inherent to Levenshtein distance and appears in any implementation. The 8-10x slowdown vs d=1 comes entirely from visiting more nodes, not from per-node inefficiency.

### Per-node cost breakdown

Measured with per-call `Clock::now()` instrumentation on 200 queries (466K nodes). Individual timings are inflated by ~20-30ns/call clock overhead, but the **ratios** between functions are reliable:

| Function | Share of cycle budget |
|----------|---------------------|
| FST node decode (read byte, memcpy offset) | ~33% |
| step() (Myers' bitvector update) | ~33% |
| can_match() (full column reconstruction) | ~33% |

No single function dominates. The three core operations split the budget roughly equally, meaning there is no isolated hotspot to optimize.

### Cache behavior

The 2MB FST data does not fit in L1 (32-48 KB) or L2 (256-512 KB) but fits in L3 (6-16 MB). However, the DFS traversal pattern provides strong locality:

- **Upper tree nodes** (depths 0-3) are revisited by every query and stay hot in L1/L2.
- **The NFA state** (`char_mask[256]` = 2KB + `LevenshteinState` = 24 bytes) fits entirely in L1.
- **DFS stack** (256 frames × ~24 bytes = ~6KB) fits in L1.

The per-node cost is **nearly identical** at d=1 (22.2 ns) and d=2 (23.8 ns), a ratio of 1.07x. If L1/L2 cache misses were significant, d=2 queries (which visit 8.5x more nodes spread across diverse FST regions) would show measurably higher per-node cost. The 1.07x ratio indicates the working set remains effectively cache-resident even under d=2 workloads.

**Platform note:** `perf stat` is not available on Windows/MSYS2. Hardware counter data (L1/L2 miss rates, instruction counts) would require Intel VTune or Windows ETW. The timing-based analysis above provides equivalent evidence: stable per-node cost across workloads with 8.5x different node visit counts is inconsistent with significant cache miss rates.

## Levenshtein DFA compilation

The Levenshtein DFA mode precompiles a deterministic automaton from the same DP recurrence as Myers' bit-parallel algorithm, but without bitvectors. Each DFA state encodes a single clamped DP column `col[0..m]`, and transitions are computed by applying the standard Levenshtein recurrence for each possible input byte.

### Why a separate Levenshtein DFA?

The existing Damerau DFA already handles Levenshtein as a special case (transpositions never fire if the state doesn't track `prev_char`). However, the Levenshtein DFA has significantly fewer states because it only needs one DP column (no `prev_col`, no `prev_char`). This means:

- **10x cheaper compilation** — Lev DFA compiles in 135 µs at d=1 vs 1,447 µs for Damerau DFA
- **Smaller transition table** — fewer states means the table fits better in L2 cache
- **Identical traversal speed** — both use the same O(1) `table[state][byte]` lookup

The Levenshtein DFA shares the same BFS construction, hash-based state deduplication, and backward-reachability pruning as the Damerau DFA, but with a simpler state representation.

## Damerau-Levenshtein via DFA compilation

The Damerau-Levenshtein implementation uses a compiled DFA rather than a bit-parallel NFA. This is necessary because the Damerau-Levenshtein recurrence requires tracking two DP columns (current and previous) plus the previous input character for the transposition check `D[i][j] = D[i-2][j-2] + 1 when query[i-1] == target[j-2] and query[i-2] == target[j-1]`.

### DFA state representation

Each DFA state encodes three components:
- `prev_col[0..m]` — the previous DP column D(i, j-1)
- `curr_col[0..m]` — the current DP column D(i, j)
- `prev_char` — the previous target character

Cell values are clamped to `max_distance + 1` to keep the state space finite. States where both columns are entirely clamped map to a dead absorbing state (state 0).

### DFA construction (BFS enumeration)

The DFA is built by BFS over reachable states. Starting from the initial state (D(i,0) = i), each state is expanded by transitioning on representative input bytes. Equivalence classes reduce the alphabet: each unique query byte is its own class, and all non-query bytes share a single representative (they never trigger transpositions since they don't appear in the query).

For each (state, byte) pair, the next state is computed using the Damerau-Levenshtein DP recurrence with transposition. States are canonicalized via a hash map. The result is a full transition table: `table[state][byte] → next_state`, enabling O(1) per-character lookup during FST traversal.

### Pruning via backward reachability

After DFA construction, a backward BFS from accepting states (where `curr_col[m] <= max_distance`) computes `can_reach_accept` for every state. During FST traversal, states where `can_reach_accept` is false are pruned immediately, avoiding dead subtree exploration.

### Performance characteristics

The DFA traversal is ~2x faster per-node than the bit-parallel Levenshtein NFA because `step()` is a single array index (`table[state][label]`) instead of ~12 bitwise operations. However, DFA compilation is expensive:

| Distance | DFA compile | States | Per-query traversal |
|----------|-------------|--------|---------------------|
| d=1 | ~1 ms | ~164 | ~23 µs |
| d=2 | ~3 ms | ~468 | ~237 µs |
| d=3 | ~9 ms | ~1,366 | ~1,066 µs |

The transition table size is `states × 256 × 4 bytes`. At d=1 with ~164 states, this is ~164 KB — fits comfortably in L2 cache. At d=3 with ~1,366 states, it grows to ~1.4 MB, which may cause L2 pressure on some architectures.

### Why not bit-parallel Damerau-Levenshtein?

Myers' bit-parallel algorithm encodes a single DP column as two bitvectors (Pv, Mv). The Damerau-Levenshtein recurrence `D[i][j] = D[i-2][j-2] + 1` requires access to the previous column's values two rows back. This cross-column, non-adjacent dependency breaks the single-column bitvector representation.

Extensions exist — Hyyrö 2003 ("A Bit-Vector Algorithm for Computing Levenshtein and Damerau Edit Distances") describes a modification of Myers' algorithm that adds ~5 bit-operations to handle transpositions by carrying the previous column's character match vector. This is a planned addition that would provide zero-startup Damerau-Levenshtein search, but it is not yet implemented. The DFA approach is used in the meantime, trading upfront compilation cost for correct and fast per-node processing.
